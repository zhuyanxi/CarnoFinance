# 期权追踪

本文将 OptionTrack 模块的需求整理为可直接录入 Jira 的 Story 草案，采用标准化的 Story 描述结构，便于产品、设计、开发、测试在同一口径下推进。

## Epic 信息

- Epic 名称: OptionTrack
- Epic 目标: 支持用户手工维护一个期权交易计划的完整生命周期，包括创建计划、记录交易、关闭计划、查看列表以及查看盈亏情况。
- 目标用户: 需要自行记录期权交易计划与执行过程的用户。
- MVP 范围: 仅支持手工录入与页面展示，不包含券商接口对接、自动下单、自动行情同步、手续费与税费精算。

## Jira Story 编写约定

每个 Story 建议至少包含以下字段:

- Summary: Jira 标题，建议直接作为工单标题。
- User Story: 采用“作为...我希望...以便...”格式。
- Business Value: 说明该 Story 的业务意义。
- Scope: 说明本 Story 负责的范围。
- Business Rules: 明确业务规则，降低实现歧义。
- Acceptance Criteria: 使用 Given / When / Then 描述可验证的验收条件。
- Dependencies: 标明对前置 Story 或公共能力的依赖。
- Out of Scope: 明确本 Story 不做的事情，避免范围膨胀。

## 通用业务假设

- 一个交易计划可以包含多个期权合约。
- 一个交易计划的状态至少包括“进行中”和“已关闭”。
- 所有计划、持仓、交易记录均由用户手工录入。
- 交易记录按“买开 / 卖开 / 买平 / 卖平”四种类型记录。
- 同一交易计划允许同时录入不同到期日、不同执行价的多个期权合约。
- 当前版本不处理自动行情拉取，不处理手续费、税费、滑点、保证金占用等衍生计算。
- 当前版本盈亏计算不纳入手续费、税费等交易成本。
- 当前版本默认只关注计划内期权合约的记录与盈亏展示，不扩展到股票现货、组合对冲腿自动关联等复杂场景。

## 技术约束与实现口径

- 存储约束: OptionTrack 使用独立 SQLite 文件 `data/OptionTrack.db`，不复用当前 `finance.db`。
- 初始化约束: 应在服务启动时自动创建 `data/` 目录和 `OptionTrack.db` 文件，并完成所需表初始化。
- 后端架构: 复用当前 Go + Gin + Bun 结构，OptionTrack API 建议统一使用 `/optiontrack` 作为路由前缀。
- UI 模块约束: `OptionTrack` 作为当前 UI 左侧导航中的独立模块入口，默认跳转到 `/optiontrack`。
- 页面路由建议: `/optiontrack` 为列表页，`/optiontrack/new` 为创建页，`/optiontrack/[id]` 为计划详情页。
- 数据建模建议: MVP 以交易台账作为唯一事实来源，S01 中录入的“建仓品种”在持久化层落为首批开仓记录，避免同时维护“初始持仓表”和“交易记录表”造成双写不一致。
- 聚合口径: 当前持仓、可平仓数量、计划关闭条件和盈亏试算均通过交易记录聚合计算，不单独维护持仓快照表。
- 金额口径: 金额由系统按“价格 x 张数 x 合约乘数”自动计算，不允许手工录入覆盖。
- 历史维护口径: 允许修改和删除历史建仓信息、交易记录，但必须同步重算持仓、计划状态和盈亏结果。
- 事务要求: 创建计划、追加交易记录、关闭计划都应放在数据库事务中执行，避免部分成功导致数据不一致。

## 公共技术前置任务

1. 在 `pkg/db/sqlite.go` 中新增 OptionTrack 专用的 SQLite 打开方法，DSN 指向 `data/OptionTrack.db`。
2. 在 `cmd/main.go` 中挂载 OptionTrack 对应的 domain 初始化和 Gin 路由注册。
3. 在 `pkg/domain` 中新增 OptionTrack 的 Bun model、DTO 和 service 方法文件，避免与现有 ETF 逻辑混杂。
4. 在 `pkg/api` 中新增 OptionTrack API handler 文件，并按 Story 拆分接口职责。
5. 在 `ui/src/lib` 中新增 OptionTrack 请求封装。
6. 在 `ui/src/routes/+layout.svelte` 左侧导航中新增 `OptionTrack` 模块入口，指向 `/optiontrack`。

## S01 创建交易计划

- Summary: [OptionTrack] 创建交易计划并录入初始建仓信息
- User Story: 作为用户，我希望手动创建一个交易计划，并录入计划名称及初始建仓品种信息，以便开始追踪一个完整的期权交易计划。
- Business Value: 没有交易计划就无法承载后续交易记录、计划状态和盈亏计算，这是整个模块的基础入口。

### Scope

- 提供“新建交易计划”入口。
- 支持录入计划名称。
- 支持录入一个或多个初始建仓品种。
- 保存成功后生成一个处于“进行中”状态的交易计划。

### Business Rules

- 计划名称为必填项。
- 至少需要录入 1 条建仓品种信息才允许保存。
- 每条建仓品种至少包含合约名称、到期日、执行价、期权方向、合约乘数、建仓时间、成本、张数。
- 张数和合约乘数必须为正整数，成本必须为正数。
- 金额由系统自动计算，不允许手工输入。
- 新建完成后的计划默认状态为“进行中”。
- 初始建仓记录的交易类型应根据用户选择明确记录为“买开”或“卖开”。
- 本 Story 只定义手工录入，不要求自动从行情或合约库补全数据。

### Acceptance Criteria

1. Given 用户进入新建交易计划页面 When 页面加载完成 Then 用户可以看到计划名称输入区和建仓品种列表录入区。
2. Given 用户未填写计划名称 When 点击保存 Then 系统阻止提交并提示计划名称为必填项。
3. Given 用户未添加任何建仓品种 When 点击保存 Then 系统阻止提交并提示至少需要录入一条建仓品种。
4. Given 用户添加了多条建仓品种 When 每条记录都填写完整且字段合法 Then 系统允许保存，并为每条记录自动计算金额。
5. Given 用户成功保存交易计划 When 保存完成 Then 系统生成一条新计划记录，状态为“进行中”，并可进入该计划详情页。
6. Given 用户输入了非法数值 When 张数或合约乘数非正整数，或成本小于等于 0 Then 系统阻止提交并给出字段级错误提示。
7. Given 用户录入多条不同到期日、不同执行价的合约 When 各条记录字段完整且合法 Then 系统允许在同一计划下保存这些合约。

### Dependencies

- 无，作为模块基础 Story 优先实现。

### Out of Scope

- 编辑既有交易计划。
- 导入外部交易记录。
- 手工覆盖系统计算出的金额。
- 自动识别合约代码或从外部行情源补全合约信息。

### Tech Sub Tasks

1. DB: 在 `pkg/db/sqlite.go` 中新增 `NewOptionTrackSqlite(ctx)`，数据库文件固定为 `data/OptionTrack.db`，并在启动时确保 `data/` 目录存在。
2. DB: 在 `pkg/domain` 中新增 `OptionTrackPlan` 和 `OptionTrackTradeRecord` 两个 Bun model，计划表至少包含 `id`、`name`、`status`、`created_at`、`closed_at`，交易表至少包含 `plan_id`、`contract_name`、`contract_code`、`expiry_date`、`strike_price`、`option_side`、`contract_multiplier`、`trade_type`、`trade_time`、`price`、`quantity`、`amount`、`entry_source`。
3. DB: 为交易表增加 `plan_id + trade_time`、`plan_id + contract_name + expiry_date + strike_price + option_side` 索引，并对 `trade_type`、`quantity`、`price`、`contract_multiplier`、`amount` 增加基础约束。
4. Backend: 在 `pkg/domain` 中实现 `CreateOptionTrackPlan`，使用事务一次性写入计划主记录和首批开仓记录，其中首批建仓记录统一落为 `trade_type=BUY_OPEN` 或 `SELL_OPEN`，金额由服务端按公式自动计算。
5. API: 在 `pkg/api` 中新增 `POST /optiontrack/plans` handler，负责参数校验、错误返回和创建成功后的计划 ID 回传。
6. Backend: 在 `cmd/main.go` 中注册 OptionTrack 初始化逻辑和创建计划接口，确保不影响现有 ETF 路由。
7. Frontend: 在 `ui/src/lib` 中新增 OptionTrack API 调用封装，并在 `ui/src/routes/optiontrack/new/+page.svelte` 中实现动态建仓品种表单，支持录入不同到期日、不同执行价的多个合约。
8. Frontend: 在创建成功后跳转到 `/optiontrack/[id]` 详情页，并在表单侧完成计划名称、数值字段、最少一条建仓记录等前端校验，同时将金额字段展示为只读自动计算值。
9. Test: 新增 domain/API 测试，覆盖空计划名、空建仓列表、非法数值、金额自动计算、事务回滚五类场景。

## S02 新增交易记录

- Summary: [OptionTrack] 在交易计划详情中新增交易记录
- User Story: 作为用户，我希望在单个交易计划详情页持续补充交易记录，以便跟踪该计划后续的开仓和平仓行为。
- Business Value: 交易计划的价值在于可持续追踪执行过程，交易记录是计划状态与盈亏计算的核心数据来源。

### Scope

- 在交易计划详情页提供新增交易记录能力。
- 支持针对计划内合约持续录入交易记录。
- 记录新增后在详情页中可查看历史交易明细。

### Business Rules

- 只有“进行中”的交易计划可以新增交易记录。
- 每条交易记录至少包含合约名称、到期日、执行价、期权方向、合约乘数、交易类型、交易价格、交易时间、张数。
- 交易类型仅支持“买开 / 卖开 / 买平 / 卖平”。
- 金额由系统自动计算，不允许手工录入。
- 平仓记录的张数不能超过该计划下对应合约维度的可平仓张数；合约维度至少包含合约名称、到期日、执行价、期权方向。
- 交易记录保存后，应更新该计划的最新交易时间与当前持仓状态。
- 允许修改和删除历史交易记录，但每次修改或删除后都必须重算当前持仓、计划状态和盈亏结果。
- 交易记录应按交易时间排序展示，默认按时间正序展示更便于复盘；如产品后续有展示要求，可再单独调整。

### Acceptance Criteria

1. Given 用户进入一个进行中的交易计划详情页 When 点击“新增交易记录” Then 系统展示交易记录录入表单。
2. Given 用户录入一条合法的买开或卖开记录 When 保存成功 Then 该记录出现在交易明细中，且计划当前持仓相应增加。
3. Given 用户录入一条合法的买平或卖平记录 When 保存成功 Then 该记录出现在交易明细中，且计划当前持仓相应减少。
4. Given 用户尝试录入买平或卖平记录 When 平仓张数大于该合约当前可平仓张数 Then 系统阻止提交并提示可平仓数量不足。
5. Given 交易计划状态为“已关闭” When 用户尝试新增交易记录 Then 系统不允许新增，并提示该计划已关闭。
6. Given 用户漏填交易类型、价格、时间、张数、到期日、执行价、期权方向或合约乘数 When 点击保存 Then 系统阻止提交并提示对应必填项。
7. Given 一个计划已有多条交易记录 When 用户进入详情页 Then 可以看到完整的交易记录列表，且排序规则一致。
8. Given 用户修改或删除一条历史交易记录 When 操作成功 Then 系统同步更新交易明细、当前持仓和相关盈亏结果。

### Dependencies

- 依赖 S01 创建交易计划。

### Out of Scope

- 批量导入交易记录。
- 自动从券商流水同步成交数据。

### Tech Sub Tasks

1. Backend: 在 `pkg/domain` 中实现 `CreateOptionTrackTrade(planID, req)`，先校验计划状态，再按合约维度聚合当前可平仓张数，最后在事务中写入交易记录，并由服务端自动计算金额。
2. Backend: 在 `pkg/domain` 中新增计划详情查询方法，返回计划基础信息、按合约维度聚合的当前持仓和交易记录列表，作为详情页与后续盈亏计算共用数据源。
3. API: 在 `pkg/api` 中新增 `POST /optiontrack/plans/:id/trades` 和 `GET /optiontrack/plans/:id` 两个接口。
4. API: 对新增交易记录接口补充字段级校验，明确仅允许 `BUY_OPEN`、`SELL_OPEN`、`BUY_CLOSE`、`SELL_CLOSE` 四种交易类型，并对关闭状态计划直接返回业务错误。
5. API: 在 `pkg/api` 中补充 `PUT /optiontrack/plans/:id/trades/:tradeId` 和 `DELETE /optiontrack/plans/:id/trades/:tradeId` 接口，支持修改和删除历史交易记录。
6. Frontend: 在 `ui/src/routes/optiontrack/[id]/+page.svelte` 中增加“新增交易记录”表单或弹窗，录入成功后刷新交易明细和当前持仓摘要，金额字段只读展示。
7. Frontend: 在详情页展示交易记录列表，默认按交易时间正序展示，并清楚标识交易类型、合约维度、价格、张数、金额和交易时间，同时提供编辑、删除入口。
8. Frontend: 在录入平仓记录时，根据详情接口返回的当前持仓做即时提示，减少用户输入超额平仓的概率。
9. Test: 增加 domain 级测试，覆盖开仓成功、部分平仓成功、超额平仓失败、修改历史后重算正确、删除历史后重算正确、已关闭计划禁止新增交易六类核心场景。

## S03 关闭交易计划

- Summary: [OptionTrack] 手动关闭交易计划
- User Story: 作为用户，我希望在交易计划已经完成并清仓后手动关闭该计划，以便将其从进行中的计划中区分出来。
- Business Value: 关闭计划可以明确计划生命周期结束，避免继续误录交易，也方便列表管理和结果复盘。

### Scope

- 提供手动关闭交易计划的操作。
- 关闭前校验计划内剩余持仓是否已经归零。
- 关闭后将计划状态更新为“已关闭”。

### Business Rules

- 只有“进行中”的计划可以关闭。
- 只有当计划下所有合约均已清仓时才允许关闭。
- 关闭行为本身不是一条交易记录，不替代平仓交易。
- 已关闭计划默认不允许继续新增交易记录。
- 若允许修改或删除历史记录，则已关闭计划在历史被改动后应重新校验是否仍满足“全部清仓”；若不满足，应阻止修改删除或要求先恢复为进行中状态，具体交互需实现时统一。
- 已关闭计划仍可被查看，用于复盘和盈亏查看。

### Acceptance Criteria

1. Given 用户进入一个进行中的交易计划详情页 When 计划所有合约当前持仓均为 0 Then 用户可以执行关闭操作。
2. Given 用户尝试关闭交易计划 When 仍存在未平仓合约 Then 系统阻止关闭，并提示存在未清仓持仓。
3. Given 用户关闭一个符合条件的交易计划 When 操作成功 Then 该计划状态更新为“已关闭”。
4. Given 交易计划已关闭 When 用户再次进入详情页 Then 系统应显示该计划已关闭的状态标识。
5. Given 交易计划已关闭 When 用户尝试新增交易记录 Then 系统阻止新增并提示计划已关闭。

### Dependencies

- 依赖 S01 创建交易计划。
- 依赖 S02 新增交易记录，用于判断当前持仓是否已清仓。

### Out of Scope

- 重新打开已关闭计划。
- 关闭后恢复交易。
- 关闭时自动生成结算单。

### Tech Sub Tasks

1. Backend: 在 `pkg/domain` 中实现 `CloseOptionTrackPlan(planID)`，关闭前统一按合约聚合当前持仓，并仅在所有持仓归零时更新计划状态。
2. Backend: 关闭计划逻辑需放在事务中执行，避免并发情况下出现“仍有持仓但已关闭”的状态错误。
3. API: 在 `pkg/api` 中新增 `POST /optiontrack/plans/:id/close` 接口，返回关闭后的计划状态和关闭时间。
4. Backend: 在计划详情查询和列表查询中补充状态字段与 `closed_at` 字段，确保 UI 可以同步展示关闭结果。
5. Frontend: 在 `ui/src/routes/optiontrack/[id]/+page.svelte` 中增加“关闭计划”按钮，仅在状态为进行中时显示，并增加二次确认。
6. Frontend: 关闭成功后刷新详情页状态、禁用新增交易入口，并显示“已关闭”标识。
7. Test: 增加测试覆盖无持仓关闭成功、存在持仓关闭失败、已关闭计划重复关闭幂等处理或错误返回、关闭后改动历史触发状态重校验四类情况。

## S04 查看交易计划列表

- Summary: [OptionTrack] 查看交易计划列表并支持分页与排序
- User Story: 作为用户，我希望查看交易计划列表，并支持分页和按创建时间排序，以便快速找到最近创建或需要关注的计划。
- Business Value: 当计划数量增加后，列表页是管理所有计划的主要入口，分页和排序是基本可用性要求。

### Scope

- 提供交易计划列表页。
- 支持分页。
- 支持按创建时间排序，默认倒序。
- 展示每个计划的基础摘要信息。

### Business Rules

- 列表默认按创建时间倒序排列。
- 列表至少展示计划名称、状态、创建时间；如已有最新交易时间字段，也建议一并展示。
- 列表应支持空状态展示。
- 分页切换后排序规则保持不变。
- 用户从列表进入任一计划详情后，应能查看该计划的完整信息。

### Acceptance Criteria

1. Given 用户进入交易计划列表页 When 页面加载完成 Then 系统展示交易计划列表，并默认按创建时间倒序排序。
2. Given 系统中存在多条交易计划 When 用户查看列表 Then 新创建的计划应出现在旧计划之前。
3. Given 列表数据超过单页承载量 When 用户切换页码 Then 系统展示对应页的数据且排序规则保持正确。
4. Given 当前没有任何交易计划 When 用户进入列表页 Then 系统展示空状态提示，而不是空白页面或报错。
5. Given 列表中存在进行中和已关闭的计划 When 用户查看列表 Then 可以清楚识别每个计划的状态。
6. Given 用户点击列表中的某个计划 When 跳转完成 Then 系统进入该计划详情页。

### Dependencies

- 依赖 S01 创建交易计划。
- 与 S03 关闭交易计划联动，用于展示计划状态。

### Out of Scope

- 按计划名称搜索。
- 按状态筛选。
- 批量操作计划。

### Tech Sub Tasks

1. API: 在 `pkg/api` 中新增 `GET /optiontrack/plans` 接口，支持 `page`、`page_size`、`sort_by=created_at`、`order=desc` 等查询参数。
2. Backend: 在 `pkg/domain` 中实现分页查询方法，返回总数、当前页数据、页码信息，并默认按 `created_at DESC` 排序。
3. Backend: 列表查询结果至少包含计划 ID、计划名称、状态、创建时间、关闭时间、最新交易时间、合约数量，供 UI 摘要展示。
4. Frontend: 在 `ui/src/routes/+layout.svelte` 左侧导航中新增 `OptionTrack` 模块入口，点击进入 `/optiontrack`。
5. Frontend: 在 `ui/src/routes/optiontrack/+page.svelte` 中实现计划列表页，包含新建计划入口、分页器、空状态和加载状态。
6. Frontend: 列表页默认按创建时间倒序展示，分页切换后保留当前排序规则，并支持跳转到计划详情页。
7. Test: 增加接口测试与页面测试，覆盖默认倒序、跨页取数、空列表展示、列表进入详情页等场景。

## S05 查看与试算交易计划盈亏

- Summary: [OptionTrack] 查看交易计划盈亏并支持手工输入当前价格试算
- User Story: 作为用户，我希望看到某个交易计划当前的盈亏情况，并能手动输入计划内持仓合约的当前价格进行试算，以便判断该计划当前表现。
- Business Value: 盈亏展示是用户最关心的结果信息，决定该模块是否真正具备复盘和辅助决策价值。

### Scope

- 在交易计划详情页展示当前盈亏情况。
- 默认基于最近一次交易记录后的持仓状态和最近成交价格口径展示盈亏。
- 支持用户手工输入当前持仓合约价格进行重新试算。
- 展示计划汇总盈亏，以及按合约拆分的盈亏明细。

### Business Rules

- 盈亏展示以交易计划中的交易记录为基础，不依赖自动行情。
- 若用户未输入当前价格，则对未平仓合约使用最近一次相关交易价格作为默认试算价格。
- 若计划已无持仓，则当前盈亏应等于已实现盈亏。
- 盈亏计算不纳入手续费、税费或其他交易成本。
- 手工输入的当前价格仅用于本次试算，不应修改历史交易记录。
- 试算结果至少应包含计划总盈亏；如存在多合约，也应支持分合约展示。
- 允许同一计划存在多个不同到期日、不同执行价的合约，盈亏需要按合约维度分别计算后再汇总。
- 本 Story 不处理手续费、税费、保证金占用、年化收益率等扩展指标。

### Acceptance Criteria

1. Given 用户进入某个交易计划详情页 When 页面加载完成 Then 系统展示该计划当前的盈亏结果。
2. Given 计划存在未平仓合约 When 用户未输入任何当前价格 Then 系统使用最近一次相关交易价格作为默认试算依据并展示结果。
3. Given 用户为一个或多个仍有持仓的合约输入当前价格 When 点击试算 Then 系统基于输入价格重新计算并展示计划总盈亏和分合约盈亏。
4. Given 用户只为部分合约输入当前价格 When 点击试算 Then 已输入的合约按手工价格计算，未输入的合约继续使用默认试算价格。
5. Given 计划已经全部清仓 When 用户查看盈亏 Then 系统展示已实现盈亏，且不要求继续输入当前价格。
6. Given 用户多次进行手工试算 When 查看交易记录 Then 历史交易数据不应被试算输入覆盖或修改。

### Dependencies

- 依赖 S01 创建交易计划。
- 依赖 S02 新增交易记录。
- 与 S03 关闭交易计划联动，用于区分已清仓计划的盈亏口径。

### Out of Scope

- 自动拉取实时行情。
- 收益率曲线。
- 手续费、税费、保证金、Greeks 等扩展计算。

### Tech Sub Tasks

1. Backend: 在 `pkg/domain` 中实现计划盈亏聚合方法，基于交易记录按合约维度计算已实现盈亏、未实现盈亏、计划总盈亏和分合约盈亏。
2. Backend: 默认试算价格口径为“每个未平仓合约最近一次相关交易价格”；若合约已清仓，则仅统计已实现盈亏；计算中不纳入交易成本。
3. API: 在 `pkg/api` 中新增 `GET /optiontrack/plans/:id/pnl` 和 `POST /optiontrack/plans/:id/pnl/preview` 接口，后者接收用户手工输入的价格覆盖值。
4. Backend: 价格试算接口只使用请求内覆盖价格参与计算，不得回写数据库，不得修改任何历史交易记录。
5. Frontend: 在 `ui/src/routes/optiontrack/[id]/+page.svelte` 中增加盈亏摘要区域，展示计划总盈亏、分合约盈亏及当前持仓。
6. Frontend: 为仍有持仓的合约提供“当前价格”输入框，用户提交后调用试算接口并刷新结果；未输入价格的合约继续使用默认口径。
7. Frontend: 对已清仓计划隐藏或禁用价格试算输入，仅展示已实现盈亏结果。
8. Test: 增加计算测试，覆盖全平仓计划、部分持仓计划、同计划多合约汇总、仅部分合约传入价格覆盖、重复试算不改写历史数据五类场景。

## Story 依赖顺序建议

1. S01 创建交易计划
2. S02 新增交易记录
3. S03 关闭交易计划
4. S04 查看交易计划列表
5. S05 查看与试算交易计划盈亏

说明:

- S04 在实现上可以与 S02、S03 并行，但如果列表页要展示状态，至少需要与 S03 的状态定义保持一致。
- S05 强依赖 S01、S02 的数据模型稳定后再进入开发更稳妥。

## 已确认产品口径

1. 交易记录明确区分为“买开 / 卖开 / 买平 / 卖平”。
2. 金额由系统按“价格 x 张数 x 合约乘数”自动计算，不允许手工覆盖。
3. 盈亏计算不纳入手续费、税费等交易成本。
4. 允许修改或删除历史建仓信息和交易记录，但需同步重算持仓、状态和盈亏。
5. 允许在同一计划内录入不同到期日、不同执行价的多个合约，并按合约维度聚合与展示。

## 通用 Definition of Done

- 页面或接口实现满足对应 Story 的验收标准。
- 必填项校验、非法数值校验、空状态展示均已覆盖。
- 计划状态流转与交易记录逻辑保持一致。
- 已完成基本测试验证，至少覆盖创建、记录、关闭、列表、盈亏试算这 5 条主链路。
